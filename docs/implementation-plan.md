# 天天象棋网页版复刻详细计划

## Summary

先做一个“网页版先行”的可玩初始版本，用 `React + TypeScript + Vite` 实现，不接 Python 后端。目标是复刻天天象棋小程序的核心体验：主页入口、象棋模式、匹配/开局、棋盘对弈、走子提示、计时、认输/超时结算、复盘回看，以及若干模式页的静态/半静态外壳。

初始版本优先保证“像、能玩、流程完整”。不做真实微信登录、在线匹配、充值、广告、商城交易、排行榜同步、真实社交关系。

## 阶段计划

### 阶段 1：项目骨架与视觉基调

- 搭建 Vite React TypeScript 项目，建立基础路由/状态结构。
- 设计整体视觉：木纹背景、棋盘材质、圆形棋子、顶部资源栏、左侧模式导航、底部主 Tab。
- 用 CSS/Canvas 绘制棋盘和棋子，不使用原版素材，避免版权风险，但尽量贴近小程序观感。
- 建立核心页面框架：首页、象棋大厅、对局页、结算页、复盘页、我的页面占位。

交付标准：打开网页后第一屏就是可操作的“天天象棋风格”主界面，不是介绍页。

### 阶段 2：象棋核心玩法闭环

- 实现标准 9x10 中国象棋棋盘、32 枚棋子、初始布局、红黑双方。
- 实现基本走子规则：车、马、相/象、仕/士、帅/将、炮、兵/卒。
- 实现选子、合法落点绿点提示、最近一步标记、吃子、将军判断、胜负判断。
- 增加简单人机对弈：玩家执红，AI 使用基础规则选择可行走法，初版不追求强棋力。
- 增加对局状态：当前回合、双方头像/名称、计时器、步时、暂停式弹窗。

交付标准：用户可以从大厅进入一局完整象棋，对弈直到将死、认输或超时。

### 阶段 3：匹配、结算与复盘

- 做“棋力评测/普通对局”入口，点击后进入仿匹配弹窗，再进入对局。
- 对局页支持认输、退出确认、超时结束。
- 结算页展示胜负、分数变化、进度条、再来一局、切换对手、复盘分析。
- 复盘页展示棋谱列表和棋盘回放控制：上一步、下一步、播放/暂停、回到开始/结束。
- 记录每一步走法，用于复盘和基础棋谱展示。

交付标准：形成“小程序式”的完整路径：首页 → 象棋 → 匹配 → 对局 → 结算 → 复盘/再来一局。

### 阶段 4：其他模式的初始外壳

- 增加“残局”页：每日残局、闯关、评分、挑战、学习等入口；先做一个可玩的简单残局。
- 增加“揭棋”页：展示评测、普通场、中级场、好友对战等入口；初版只做规则说明和静态棋盘预览。
- 增加“更多玩法”页：翻翻棋、五子棋、敬请期待；五子棋可做简单本地人机或静态入口。
- 增加“排位赛”页：段位、赛季、5/10/20 分钟场入口；进入后仍复用象棋核心对局。
- 增加“学棋 / 棋界 / 发现 / 我”的主要页面外壳，复刻入口密度和信息布局。

交付标准：用户点开主要 Tab 和主要模式时，不会出现空白页面；核心可玩内容仍集中在象棋。

### 阶段 5：打磨与验收

- 检查移动端和桌面端布局，优先适配手机竖屏和桌面浏览器。
- 修正文字溢出、按钮错位、棋盘比例失衡、弹窗遮挡等视觉问题。
- 用浏览器实际试玩核心路径，截图检查棋盘、主页、结算、复盘。
- 添加核心规则测试：各棋子合法走法、吃子、将军、胜负、复盘步进。
- 优化简单 AI，避免明显非法走子或长时间卡住。

交付标准：初始版本可以稳定运行，核心流程连续试玩无阻断。

## 初始版本后的后续计划

### V1.1：玩法增强

- 增强残局系统：多关卡、提示、失败重试、评分。
- 实现更完整的揭棋规则，而不是只做入口和预览。
- 五子棋做成完整本地玩法，支持人机和双人。
- 增加棋谱导出/导入，支持复制简易棋谱文本。

### V1.2：棋力与 AI

- 引入更强的象棋 AI 或搜索算法，例如基础 minimax + alpha-beta。
- 增加难度等级：新手、普通、高手。
- 增加局面评估：子力价值、将军威胁、吃子优先级。
- 增加“棋力评测”结果页，让对局结果影响本地段位/积分。

### V1.3：账号与数据

- 根据需要再加入 Python 后端，负责用户、本地/云端存档、排行榜、历史棋谱。
- 后端建议使用 `FastAPI + SQLite/PostgreSQL`。
- 前端增加登录态、个人主页、战绩统计、收藏棋谱。
- 所有网络功能保持可选，不影响单机游玩。

### V1.4：在线对战

- 增加 WebSocket 实时对战。
- 支持房间码好友对战、匹配队列、掉线重连、观战。
- 增加聊天、表情、悔棋/和棋请求。
- 服务端校验走子，避免客户端作弊。

### V1.5：商业化与内容系统

- 增加商城、背包、棋盘皮肤、棋子皮肤，但只使用自制素材。
- 增加活动页、任务系统、签到奖励。
- 增加课程/题库后台数据结构。
- 如果需要做成小程序，再评估迁移到 Taro/uni-app 或重写微信小程序版本。

## Public APIs / Interfaces

- 前端内部需要定义清晰的棋局模型：棋子、坐标、走法、对局状态、计时状态、结算结果、复盘记录。
- 初始版本不提供外部 HTTP API。
- 如果后续加入 Python 后端，再新增接口：用户资料、对局记录、排行榜、在线房间、棋谱存储。
- 棋盘组件需要保持可复用，供普通对局、残局、复盘、揭棋预览共用。

## Test Plan

- 视觉测试：主页、象棋大厅、对局页、结算页、复盘页在桌面和移动视口下不溢出、不遮挡。
- 规则测试：每种棋子的合法走法、非法走法、吃子、将军、将死、胜负判断。
- 流程测试：首页进入对局、完成对局、进入结算、复盘、再来一局。
- 计时测试：步时减少、超时结算、认输结算。
- 回归测试：走子后棋盘状态、历史记录和复盘状态一致。

## Assumptions

- 初始版本以网页为目标，不直接做微信小程序。
- 初始版本不使用 Python；Python 留到后端、AI、数据存储阶段再加入。
- 初始版本不接真实账号、支付、广告、社交和在线匹配。
- 视觉上追求接近天天象棋的体验，但不复制原版图片、商标或专有素材。
- 优先级为：标准象棋核心闭环 > 结算复盘 > 主导航页面 > 残局/揭棋/五子棋完整规则。
