# 三 Agent 并行实现象棋 / 揭棋 / 残局强规则补强计划

## Summary

- 先串行使用 Computer Use 试玩微信天天象棋三个模块，顺序为：象棋 → 揭棋 → 残局，避免多个 agent 同时操作微信导致闪退。
- 试玩时可以大胆进入对局、匹配、重开、认输、使用提示、消耗普通游戏币或道具；唯一硬限制是不触碰支付、充值、实名认证、真实购买等入口。
- 观察重点是玩法规则、局内流程、胜负/结算、入口参数和实际手感，不追求逐像素 UI 复刻。
- 观察完成后并行启动 3 个 worker agent，分别负责象棋、揭棋、残局；主线程统一整合、验收和修冲突。

## Key Changes

- 先建立共享游戏内核边界，避免三个 agent 同时大改同一块逻辑：
  - 公共棋盘、棋子、坐标、走法类型继续复用现有 `src/game.ts`。
  - 将强规则逻辑拆成标准象棋、揭棋、残局题目与判题三类能力。
  - UI 层只调用模块 API，不在组件里堆复杂规则分支。
- Agent 1：象棋模块
  - 补强标准象棋规则、将军/应将/将死、AI 合法走子与结算原因。
  - 保证大厅 → 匹配 → 对局 → 结算 → 复盘路径不退化。
- Agent 2：揭棋模块
  - 实现暗子身份、首次移动揭示、真实棋种合法走法、揭棋专属计时/结算文案。
  - 不做真实在线场、财富结算、好友房间。
- Agent 3：残局模块
  - 实现可扩展题目结构、目标判定、提示、失败/成功状态、重来与计时步数。
  - 优先规则正确和判题闭环，不扩展评论、排行、分享等在线功能。

## Computer Use Observation Plan

- 每个模块单独试玩并记录，主线程操作，worker 不同时控制微信。
- 象棋：大胆走完整局部流程，包括匹配、开局说明、走子、菜单、认输/超时或结算、复盘。
- 揭棋：进入评测或可用场次，观察暗子显示、明示棋子、暗子选中/移动/揭示时机、AI 应手和退出确认。
- 残局：进入每日残局或闯关，尝试提示、正确走法、错误走法、重来、退出和评论/评级入口。
- 避免操作：支付、充值、购买、实名认证、绑定账号、任何需要真实付款或授权敏感信息的入口。
- 若微信闪退或无法进入某模块，以仓库现有观察文档和本轮可观察到的部分为准。

## Test Plan

- 单元测试：
  - 标准象棋：各棋子合法走法、将军过滤、将死、吃将、AI 不走非法步。
  - 揭棋：暗子按真实身份生成合法落点，首次移动后揭示，揭示后标签保持真实棋种。
  - 残局：正确解成功、错误走法反馈、重置恢复初始局面。
- 流程测试：
  - 象棋完整路径：大厅 → 匹配 → 开局 → 走子 → AI 应手 → 认输/结束 → 结算 → 复盘。
  - 揭棋路径：入口 → 暗子选择 → 移动揭示 → AI 应手 → 重开。
  - 残局路径：入口 → 提示 → 正确走法 → 成功 → 重来。
- 验收：
  - `npm test`
  - `npm run build`
  - 桌面与手机视口各跑一遍核心路径，确认无空白、遮挡、文字溢出。

## Assumptions

- 本轮目标是“强玩法规则”，UI 做必要配套，不做完整商业化、社交、广告、充值、真实匹配。
- 试玩期间允许消耗普通游戏币、门票、提示类资源，只要不涉及支付。
- 微信观察若与现有文档冲突，以本轮实际试玩为准；观察不到的细节采用仓库文档默认值。
- 三个 worker 可以并行实现，但必须有明确模块 ownership；最终由主线程统一 merge、修冲突、补测试。
